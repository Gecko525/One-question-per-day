hash和history都属于浏览器自身的特性，都是监听url的变化来匹配路由规则展示相应的页面。目前前端路由（改变视图不会发送请求）就这两种方式。 1.url地址栏有#号。 2.#前面的url会被包含在请求中。而改变#后面的值，浏览器实现组件渲染，只是对浏览器的指导，对后端完全没有影响。所有不会重新加载页面当然也不会出现404。 3.hash模式背后的原理是onhashchange事件。window.onhashchange = function() { console.log(location.hash) //比如http://www.abc.com/#/index hash值为#/index } 1.地址没有#号。 2.利用h5的pushState()方法和replaceState()方法来对浏览器的浏览记录进行修改。利用state对象的状态保存来达到不用请求后端来渲染的效果。 3.刷新页面或者手动输入 URL 后回车会向服务器发送请求。所以要配合后台做重定向。 4.pushState可以设置新的url和当前的url一样，也会把记录保存到栈中。而hash则必须和当前#后面不一样的值时才能把记录存到栈中。 1.hash模式下，仅#号之前的地址才会包含在请求中。因此即使后端没有做到对路由的全覆盖也不会出现404。 2.history模式下url的地址必须要和实际项目配置的后端发送的地址相同。否则404。 3.#号
